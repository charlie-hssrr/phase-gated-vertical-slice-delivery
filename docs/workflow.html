§ 01

Core philosophy

Four ideas the whole method rests on
1.1

Build in phases, not feature piles

A project is not one large backlog. Separate foundational modelling, product completion, and hardening into distinct passes so they never collide in a single change.

1.2

Build slices, not layers

A slice cuts vertically through data model, service, UI, tests and telemetry — but solves exactly one defined problem. No horizontal "build all backend models".

1.3

Invariants before features

Every project has rules that must never break — tenant boundaries, ownership scope, provenance. Name them early, enforce them in the correct layer, review them at every slice.

1.4

Non-goals are a delivery tool

Every phase and slice states what it is not doing. Explicit out-of-scope text is what stops scope expansion, speculative abstraction and cleanup rabbit-holes.

§ 02

The phase model

A → B → C · each separated by an evidence gate
A → B
B → C
A
Foundation

Structural correctness

“Is the system structurally correct?”

  • Define entities, relationships & ownership
  • Introduce core contracts and skeletons
  • Enforce or explicitly track invariants
  • Record high-risk architecture decisions
Done whencore model is coherent, boundaries enforced or gated, nothing regressed, B can start without guessing.
B
Completion

Product coherence

“Does it work as a coherent product?”

  • Close missing product loops
  • Add management, search, retrieval, export
  • Replace frontend-only conventions with contracts
  • Resolve UX contradictions
Done whenthe primary flow works end-to-end and can be demoed without explaining internal gaps.
C
Hardening

Production readiness

“Can this run safely in production?”

  • Tighten security & failure handling
  • Add observability, metrics, runbooks
  • Remove temporary compatibility paths
  • Cover edge cases with tests
Done whenflows are reliable, failures handled deliberately, the release / production gate is passed.
◂ ordered, not one-way

The order is a default, not a ban on going back. A gate stops you moving forward on optimism — it never forbids moving back on evidence. A foundation gap found in B becomes a new A-slice, recorded in the decision log — never folded silently into unrelated work.

§ 05

The gate model

A decision point that separates optimism from evidence
Hard blocker

Stops the next phase

The next phase cannot start until this is resolved. No exceptions, no "we'll get to it".

Completion blocker

Stops phase close

The next phase can start, but the current phase cannot be called complete until this is resolved.

Deferred / accepted

Carried forward

A known issue deliberately tolerated — allowed only if it is explicitly named and signed off.

A gate must be  evidence-based
Unknown  is not pass
Deferred is allowed  only if named
Separate start blockers from  completion blockers
Process visibility ≠  technical blocker
Never hide a technical blocker as  a process issue

§ 08

Right-sizing the method

Dial the paperwork to the risk — not every slice needs all of it

The usual way a process like this fails is not that people ignore it — it is that they apply all of it to work that never needed all of it, the overhead drowns the value, and the whole thing gets dropped as bureaucracy. The phases, slices, invariants and gates scale. The artefacts are what you turn up and down.

Minimum
viable loop
  1. State the one outcome + what's out of scope
  2. Make the change narrowly
  3. Show evidence it works — a test, a run, a check
  4. Review the diff before it lands
Low risk
Copy change, isolated helper, additive UI state, low-traffic internal tool.
scope line + non-goals
a test or manual check
diff review
Standard
New behaviour, a new product loop, a contract used in one place.
slice spec
tests in the same change
decision log if non-obvious
diff review
High risk
Security / ownership boundary, multi-tenant data path, multi-caller API, migration, hard to roll back.
full slice spec
negative + boundary tests
callsite enumeration
decision log + gate entry
manual validation log
The artefact has to buy down a real risk on this slice. If it doesn't, skip it — and say you skipped it.

Use less when

  • the project is small, short-lived or low-stakes
  • one builder, fast feedback loop
  • mistakes are cheap and easily reversed
  • you're prototyping and the model is meant to move

Use the full weight when

  • multi-tenant or security-sensitive systems
  • shared platforms other teams depend on
  • migrations and hard-to-reverse blast radius
  • work handed between people, or AI-driven over many slices

Running light is a legitimate choice — but it's a decision, not an accident. Record it in one line so the next person knows corners weren't cut by mistake.

Common failure modes

What the gates and slices are defending against
Symptom

FixEnforce the Phase A gate; record unresolved architecture decisions before moving on.
Symptom

FixMove the invariant into a backend or shared domain service.
Symptom

FixRequire behaviour tests and manual checks for behaviour-changing slices.

FixLog cleanup as carry-forward unless it is directly required by the slice.
§ 11

Influences & prior art

A synthesis, not an invention — here's the lineage
Stage-GateRobert Cooper
Explicit phases separated by evidence-based go / no-go gates. The phase-gated half of the name.
Vertical Slice ArchitectureJimmy Bogard
Build outcomes that cut through every layer, not horizontal layers. The vertical slice half of the name.
Walking Skeleton / Tracer BulletsCockburn · Hunt & Thomas
A minimal end-to-end path first, then flesh it out. Phase A is essentially a walking skeleton.
Domain-Driven DesignEric Evans
Entities, ownership boundaries and invariants as first-class design concerns.
Design by ContractBertrand Meyer
Invariants as rules that must hold — enforced in the correct layer.
Architecture Decision RecordsMichael Nygard
The decision log — capturing why a choice was made, when it was made.
Make it work / right / fastKent Beck
The shape of A → B → C: correctness, then completeness, then hardening.
Shape UpBasecamp · Ryan Singer
Fixed appetite, aggressive scope-hammering, explicit no-gos. The closest philosophical sibling.
Lean / small-batchToyota Production System
Frequent, small, controlled increments over large batches.
Secure-by-designthreat modelling
No implicit trust, no silent defaults, negative and boundary tests on sensitive paths.

Where it diverges from Shape Up: Shape Up deliberately avoids heavy tracking and gates — it bets on a fixed appetite and trusts a small team. This method goes the other way and adds gates and an evidence trail, because it targets systems where ownership boundaries matter as much as features and where an AI agent is in the loop. On lower-risk work, right-sizing brings it back toward that lightness.

The individual pieces are well-established. The contribution is the synthesis and its target — pointing decades-old discipline at one problem: keeping delivery controlled when an AI agent is doing the typing.

The discipline experienced engineers apply by instinct, written down as guardrails an AI agent has to pass through — turning fast AI-assisted implementation into controlled delivery.
